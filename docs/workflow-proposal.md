# Workflow & Pipeline Proposal

## Overview

Workflows are the core orchestration primitive in Orca. A workflow defines how agents and tasks connect into an execution graph. Under the hood, a workflow compiles to a LangGraph `StateGraph`.

## Workflow syntax

### Linear flow

```hcl
workflow simple {
  researcher -> writer -> reviewer
}
```

Compiles to:
```
START -> researcher -> writer -> reviewer -> END
```

### Branching (conditional edges)

```hcl
workflow support {
  classifier -> {
    urgent:  escalator -> human
    normal:  auto_reply
    spam:    archive
  }
}
```

The classifier agent returns a label. The workflow routes to the matching branch. Each branch can be a single node or a chain.

Compiles to a LangGraph conditional edge:

```python
def route_after_classifier(state):
    return state["classification"]

graph.add_conditional_edges("classifier", route_after_classifier, {
    "urgent":  "escalator",
    "normal":  "auto_reply",
    "spam":    "archive",
})
```

### Parallel execution

```hcl
workflow gather {
  [researcher_a, researcher_b, researcher_c] -> aggregator
}
```

Square brackets = parallel. All three run concurrently, results merge into aggregator.

### Loops / feedback

```hcl
workflow iterative_writing {
  writer -> reviewer -> {
    approved: done
    revise:   writer  # loops back
  }
}
```

The `done` keyword is a built-in terminal node (maps to `END`).

### Nested workflows

```hcl
workflow inner_review {
  writer -> reviewer
}

workflow outer {
  researcher -> inner_review -> publisher
}
```

A workflow can reference another workflow as a node. This compiles to a LangGraph subgraph.

## Workflow properties

```hcl
workflow report_pipeline {
  researcher -> writer -> reviewer

  # Error handling
  on_error = retry(3)        # retry failed node up to 3 times
  # on_error = fallback(alt) # route to fallback agent
  # on_error = stop          # halt execution

  # Timeout
  timeout = "5m"

  # State schema (optional, for advanced use)
  state = {
    topic       = "string"
    findings    = "list"
    report      = "string"
    approved    = "bool"
  }
}
```

## Pipeline shorthand

A pipeline is NOT a separate concept — it's just a linear workflow. No need for a `pipeline` keyword. This keeps the language smaller:

```hcl
# This IS a pipeline. No special syntax needed.
workflow etl {
  extractor -> transformer -> loader
}
```

If we find users commonly want a shorter form, we could add `pipeline` as an alias later. But for now, one keyword (`workflow`) is simpler.

## Data flow between nodes

By default, each node receives the full workflow state and can read/write to it. This mirrors LangGraph's `State` pattern.

For explicit data passing (optional, advanced):

```hcl
workflow explicit_data {
  researcher -> writer {
    pass = [findings, sources]  # only pass these fields
  } -> reviewer
}
```

## Compilation strategy

Each workflow compiles to:

1. A Python `TypedDict` or `dataclass` for state
2. Node functions wrapping each agent/task
3. A `StateGraph` with edges matching the `->` chains
4. Conditional edges for `{ label: node }` blocks
5. `START` and `END` sentinel connections

### Source mapping

Every generated line includes a comment with the source `.oc` location:

```python
graph.add_node("researcher", researcher_node)  # research.oc:42:3
graph.add_edge("researcher", "writer")          # research.oc:45:3
```

This enables stack traces from Python to map back to Orca source.

## Full example with generated output

### Input: `agents.oc`

```hcl
model gpt4 {
  provider    = "openai"
  version     = "gpt-4o"
  temperature = 0.2
}

agent researcher {
  model  = gpt4
  tools  = [web_search]
  prompt = "Research the given topic thoroughly."
}

agent writer {
  model  = gpt4
  prompt = "Write a clear report from the research findings."
}

workflow report {
  researcher -> writer
}
```

### Output: `build/report_workflow.py`

```python

    """Research the given topic thoroughly."""
    # ... tool-bound agent invocation
    pass
```

## Open questions

1. **State inference**: Should Orca infer the state schema from agent outputs, or require explicit `state = {}` blocks?
2. **Human-in-the-loop**: How to express "pause here and wait for human input"? Possible syntax: `-> human_input ->` as a built-in node.
3. **Checkpointing**: LangGraph supports persistence/checkpointing. Should Orca expose this? E.g., `checkpoint = "sqlite"` on the workflow.
